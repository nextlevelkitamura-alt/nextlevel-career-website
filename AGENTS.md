# NextLevel Career Site - プロジェクトルール

## 正本とメイン作業場所

- このファイル（ルートの `AGENTS.md`）をAI作業指示の正本にする。
- 通常の開発作業は、既存のローカル `main` worktree をメイン作業場所として進める。
- `worktree` / feature branch / fix branch は、明示された並行実装、大型機能、DB migration、本番反映保留、未コミット差分の保護が必要な場合だけ使う。
- `main` 以外のbranchまたはworktreeで作業している場合は、作業開始時と完了時にbranch名・worktree path・`local main` への統合状況を報告する。
- 完了扱いの基準は、ユーザーが明示的に保留しない限り、対象変更が `local main` に取り込み済みであること。
- 作業完了時は、ユーザーが明示的に不要と言わない限り、対象ファイルだけをstageして必ずコミットする。
- コミットは大きく溜めず、動作確認できる小さな完了単位でこまめに作る。
- push / deploy は別ゲート。完了報告では `local main` / `origin/main` / production の反映状態を分けて書く。

## ドキュメント構造

| ディレクトリ | 役割 | AI操作権限 |
|-------------|------|-----------|
| `docs/ROADMAP.md` | 3ヶ月ロードマップ | 読む。変更時は即コミット&プッシュ |
| `docs/企画/` | 企画・アイデア | 新規作成OK。**実装は不可** |
| `docs/計画/進行中/` | 進行中の計画 | 実装対象。完了時は 完了/ に移動 |
| `docs/計画/未着手/` | 承認済み未着手 | 読むだけ。着手時に 進行中/ に移動 |
| `docs/計画/完了/` | 完了した計画 | 読むだけ |
| `docs/仕様書/` | 仕様書 | 参照用。変更時は確認を求める |
| `docs/インフラ/` | インフラ・環境設定 | 参照用 |
| `docs/アーカイブ/` | 過去の記録 | 読むだけ |

## 企画ライフサイクル

```
1. アイデア   → docs/企画/YYYYMMDD-slug.md (状態: アイデア)
2. 提案       → 仕様・工数を追記 (状態: 提案)
3. 採用       → 企画/採用/ に移動、ROADMAP.md に週割当
4. 不採用     → 企画/不採用/ に移動（理由記録）
5. 計画化     → 計画/未着手/ または 計画/進行中/ に移動
6. 実装       → 計画/進行中/ で作業 (状態: 進行中)
7. 完了       → 計画/完了/ に移動 (状態: 完了)
```

## フロントマター状態値の一覧

| 状態 | 意味 | 使用場所 |
|------|------|---------|
| アイデア | 思いつき段階 | 企画/ |
| 提案 | 仕様を書いた段階 | 企画/ |
| 採用 | 採用決定 | 企画/採用/ |
| 不採用 | 不採用（理由つき） | 企画/不採用/ |

| ブロック中 | ブロッカーあり | 計画/進行中/ |

## ROADMAP.md 変更ルール

変更したら即座に実行:
```bash
git pull && git add docs/ROADMAP.md && git commit -m "ROADMAP: {変更内容}" && git push
```

## 作業開始時の読み込み順

1. `AGENTS.md`（このファイル）
2. `docs/ROADMAP.md`（全体計画）
3. `docs/計画/進行中/`（進行中タスク）
4. `git status --short --branch`（既存差分と作業branch確認）

## 作業フォーカス

- 実装してよい主対象は `docs/計画/進行中/` の計画に紐づくもの。
- `docs/企画/` はアイデア置き場なので、計画化されるまで実装しない。
- `docs/計画/未着手/` は読むだけ。着手する場合は `docs/計画/進行中/` に移してから作業する。
- 複数の進行中タスクがある場合は、ユーザーが指定したタスクを優先する。指定がなければ優先度「高」と収益化・本業直結のものを優先する。

## 開発コマンド

- 開発サーバー: `npm run dev`（`http://localhost:3003`）
- 外部端末確認用: `npm run dev:host`
- lint: `npm run lint`
- test: `npm test`
- 型確認: `npx tsc --noEmit`
- build: `npm run build`
- Cloud Runデプロイ: `npm run deploy`（明示指示がある場合だけ）

## コーディング規約

- **言語**: TypeScript（日本語コメント・コミットメッセージ）
- **フレームワーク**: Next.js 14 (App Router), Tailwind CSS
- **DB**: Supabase (PostgreSQL)
- **AI**: Google Gemini 2.0 Flash
- **メール**: Resend
- **ホスティング**: Google Cloud Run
- App RouterではServer Component / Server Actionを優先し、Client Componentは状態管理・ブラウザAPI・操作UIが必要な範囲に限定する。
- UIは既存のTailwind設計と `lucide-react` を優先し、不要な新規UIライブラリを追加しない。
- DB変更は必ず `supabase/migrations/` にmigrationとして残し、手動SQLだけで済ませない。

## 検証ルール

- 小さなUI/文言変更: 対象画面の目視確認、または該当コンポーネントの影響範囲確認。
- TypeScript/ロジック変更: `npx tsc --noEmit` と必要な単体テストを実行する。
- Server Action、認証、DB、外部連携変更: `npm run build` か、少なくとも関連テストと型確認を実行する。
- 検証できなかった場合は、理由と未検証リスクを完了報告に書く。

## AI作業ルーティング

- 非自明な実装依頼は、作業前に目的・対象ファイル・検証方法を短く整理する。
- 要件、進捗、矛盾、PR順、デプロイ順、AGENTS/CLAUDE同期は `kaihatsu-kanri` の観点で管理する。
- 複数agentやworktreeに分ける判断は、競合しやすいファイル、DB migration、デプロイ順、未コミット差分を確認してから行う。
- `AGENTS.md` と `CLAUDE.md` に共通指示を二重管理しない。共通指示はこのファイルに集約する。

## 禁止事項

- `.env` ファイルの出力・コミット
- 破壊的変更の無断実行
- `企画/` 内のアイデアの勝手な実装
- `docs/ROADMAP.md` の変更後にコミット・プッシュを忘れること

## 完了報告

- 変更した主なファイルを列挙する。
- 実行した検証コマンドと結果を書く。
- 実行していない検証があれば理由を書く。
- branch / worktree を使った場合は、`local main` への統合状態、`origin/main` へのpush状態、production反映状態を分けて書く。
